# v9 Hands, Memory, and a Pulse — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Behavior contracts + test specs, no reference code. Three tasks in order.

**Goal:** Close the loop three ways: `fix` turns a named violation into an applied patch (the tool grows hands), `blame` finds the commit that broke the layout (memory), `watch` streams live diffs as the dev server hot-reloads (a pulse).

## Global Constraints

- Evidence: FOREGROUND `npm run typecheck && npx vitest run && npm run build` with exit codes checked. 175/175 green at base `7102690`. RED-first.
- No new deps; additive contracts; determinism where promised; alchemist commits.
- HYGIENE (hard rules learned the hard way): never put the literal leak-test pattern in a compound shell command; any orphan pkill runs standalone; test-spawned MCP servers use `node --import tsx` single-process; run suites in the foreground.
- SAFETY (new for v9, non-negotiable): `fix` NEVER writes without `--apply`; `fix --apply` refuses when the target file's content at the expected location doesn't match the suspect declaration (stale/mismatched source guard) and refuses outside `--root`; `blame` NEVER touches the user's working tree, index, or HEAD (temp `git worktree add` only, always cleaned up — including on error paths); `watch` respects SIGINT/SIGTERM with a clean Chrome shutdown.

### Task 1: `bettercss fix` — mechanical patches for fixable violations

**Files:** New `src/core/fix.ts`. Modify `src/cli.ts` (command + USAGE), `src/mcp.ts` (9th tool), README (row + a Safety note). Fixture: `fixtures/fixable/` (a page with one of each fixable violation, styled from a local `styles.css`). Tests: `test/fix.test.ts` + skin tests.

**Contract:**
1. `bettercss fix <url> --root DIR [--apply] [--selector S] [+ the usual viewport/state flags check accepts]`. Default is DRY-RUN: prints proposed patches as unified-diff-style hunks with `file:line`, applies nothing. MCP tool `fix` mirrors (apply boolean, root string, both required for writes; description states plainly that apply EDITS FILES).
2. Fixable rules v1 (FIXABLE set — everything else reports `no mechanical fix for <rule> — see suspect`):
   - `text-clip` → add `text-overflow: ellipsis;` to the clipped element's overflow rule (the suspect rule's block).
   - `tap-target` → add `min-width: 24px; min-height: 24px;` to the suspect rule's block.
   - `viewport-overflow`/`parent-bleed` where the suspect is a fixed px `width` → replace with `max-width: 100%` PLUS keep the original as a comment (`/* was: width: 1400px */`). If the suspect isn't a px width, report no-mechanical-fix.
3. URL→file mapping: the suspect's stylesheet URL pathname resolves against `--root` (e.g. `http://host/css/app.css` + root → `<root>/css/app.css`). Source-mapped suspects resolve via the map's `sources` path against root. If the resolved file doesn't exist → clear error naming both the URL and the attempted path.
4. Stale-source guard: before writing, read the resolved file and verify the suspect declaration text appears at/near the expected line (±3 lines tolerance for concurrent edits); mismatch → refuse with a clear error, apply nothing for that patch (other patches in the same run may still apply; say which did/didn't).
5. `--apply` then automatically re-runs `check` (same url/flags) and prints `before: N violations → after: M` plus any NEW violations the patch introduced (regression honesty). Exit 0 only if M < N and no new violations; else exit 1.
6. Inline `<style>` suspects (file `<style>` or the page URL): v1 refuses with `inline styles not patchable — edit <page>:<line>`; document.

**RED tests:** (a) dry-run on the fixable fixture prints a patch per fixable violation with correct file:line and applies nothing (file mtime/content unchanged); (b) `--apply` on a temp COPY of the fixture fixes text-clip (re-check shows the violation gone, css file contains ellipsis); (c) stale-source guard: mutate the css copy before apply → refusal message, file unchanged; (d) missing --root on apply → exit 2; (e) px-width bleed patch produces max-width:100% + comment and after-check improves; (f) MCP fix with apply=false returns patches, never writes.

**Commit:** `Added fix command proposing and applying mechanical patches`

### Task 2: `bettercss blame` — which commit broke the layout

**Files:** New `src/core/blame.ts`. Modify `src/cli.ts`, `src/mcp.ts` (10th tool), README. Tests: `test/blame.test.ts` (builds a THROWAWAY git repo in a temp dir with scripted commits — never the project repo).

**Contract:**
1. `bettercss blame --root DIR --page REL.html [--selector S] [--max-commits N (default 25)] [--viewport WxH]`. Scope v1: STATIC roots only — each historical version is served with the built-in static-server machinery from a temp worktree. Dev-server/build-step projects are out of scope (documented; a `--serve CMD` hook is the named future path).
2. Semantics: determine the CURRENT bad state = the set of check violations at HEAD's working tree for `--page` (optionally filtered to `--selector`'s subtree). If HEAD is clean → `nothing to blame — page is clean` exit 0. Then walk commits BACKWARDS from HEAD (linear, newest→oldest, capped at max-commits; linear beats bisect because layout states can flicker): for each commit, `git worktree add` a temp checkout, serve it, run the same check, compare. First commit (going back) where the state is GOOD marks the culprit as the commit AFTER it (the newest bad one following a good one).
3. Output: `broken by <short-sha> "<subject>" (<relative-date>, <author>)` then the layout delta between good and bad commits (reuse diffTrees against the two worktree renders) and the violations introduced. If all N commits are bad: `still broken <N> commits back — raise --max-commits`. Exit 1 when a culprit is found (it's a regression report), 0 when clean.
4. Repo safety (hard): all checkouts via `git worktree add --detach` into the scratch temp dir; worktrees removed + pruned in a finally; the user's HEAD/index/tree NEVER change. The throwaway-repo test asserts the main repo's HEAD and status are untouched after a run.
5. MCP tool `blame` mirrors (root, page, selector?, maxCommits?, viewport?).

**RED tests:** (a) throwaway repo: commit1 good page → commit2 unrelated change → commit3 introduces a px-width bleed → commit4 unrelated; blame names commit3 with subject/author, exit 1; (b) clean HEAD → `nothing to blame`, exit 0; (c) all-bad within cap → the raise-max-commits message; (d) safety assertion per contract 4; (e) MCP mirror of (a).

**Commit:** `Added blame command finding the layout-breaking commit`

### Task 3: `bettercss watch` — live HMR diff stream

**Files:** New `src/core/watch.ts`. Modify `src/cli.ts`, README, skill (watch enters the doctrine: "keep watch running while editing; read its stream instead of re-running diff"). CLI ONLY in v1 — a streaming daemon doesn't fit MCP request/response; document that agents drive it via a background shell. Tests: `test/watch.test.ts` (subprocess-driven against a mutable temp fixture copy).

**Contract:**
1. `bettercss watch <url> [--viewport WxH] [--interval MS (default 500)]`. Holds ONE page open (long-lived variant of the withPage internals — implementer's judgment on the seam; Chrome shutdown on exit paths). On start: prints the initial `check` summary + `watching <url> (Ctrl+C to stop)`.
2. Loop: every interval, read the settle-signature hash (reuse the existing in-page signature). On change: wait for settle (existing machinery, existing cap/note), then (a) diffTrees vs the previous stable tree → print delta lines, (b) re-run invariants → print NEW violations and RESOLVED violations (deltas only, grouped as usual), (c) update the baseline. Quiet when nothing changes (no heartbeat spam).
3. Full page reloads (HMR full-refresh / dev-server restart): detected via frameNavigated to the same URL → re-arm and continue with a `page reloaded` line (NOT an error — watch's whole job is surviving reloads; navigation to a DIFFERENT url → `navigated away to <url> — stopping` exit 1).
4. Output blocks are timestamped `[HH:MM:SS]` (wall-clock is fine here — watch is explicitly an observation stream, not a deterministic snapshot; the DIFF CONTENT stays deterministic).
5. SIGINT/SIGTERM → clean shutdown (Chrome down, exit 0). If the dev server dies mid-watch → `page unreachable — stopping` exit 1 (no infinite retry in v1).

**RED tests:** (a) start watch on a temp-served fixture copy, edit the css file (move an element), assert the stream (within a generous timeout) emits a `moved` line for that element, then SIGINT and assert exit 0 and no Chrome orphans; (b) edit that introduces a violation → stream emits `new violation: parent-bleed ...`; then revert the edit → stream emits `resolved:`; (c) kill the fixture server → watch exits 1 with the unreachable message. Timing-sensitive: assert substrings with generous timeouts, never exact timings.

**Commit:** `Added watch command streaming live layout diffs`
