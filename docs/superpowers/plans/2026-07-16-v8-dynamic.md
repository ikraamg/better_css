# v8 Dynamic Layout Truth Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Behavior contracts + test specs, no reference code. Three tasks, strictly in order (each builds on the last).

**Goal:** Extend ground truth beyond the frozen post-load moment: (1) layouts reachable only through real input events (menus, modals, scroll-dependent UI), (2) deterministic capture of animated/transitioning pages, (3) load-time layout stability (what shifted while loading). The thesis is unchanged — dynamic ≠ nondeterministic: frozen time and settled states are as deterministic as what ships today.

### Task 1: Interaction pre-steps — `--click` and `--scroll-to`

**Files:** New `src/core/interact.ts` (one responsibility: run ordered pre-steps + settle). Modify `src/cli.ts`, `src/mcp.ts`. Fixture: `fixtures/interactive/index.html`. Tests: new `test/interact.test.ts` + CLI/MCP additions.

**Contract:**
1. CLI: `--click S` (repeatable — extend `flags()` so repeated occurrences of designated flags accumulate into arrays; all other flags keep last-wins, byte-identical) and `--scroll-to S_or_Y` (selector or pixel y). MCP: `click?: string[]`, `scrollTo?: string` on layout/check/inspect/explain/verify (NOT snapshot/diff in v1 — same stale-state reasoning as pseudo-states; still rejected loudly).
2. Execution order: navigate → scrollTo (if given) → clicks in argument order → settle → (existing state forcing if given) → capture. Document the order in tool descriptions.
3. Clicks are REAL trusted events: resolve the selector (existing suggestions error on no match), get the element's center via `DOM.getBoxModel` (or getContentQuads), `Input.dispatchMouseEvent` mousePressed+mouseReleased. If the click triggers a navigation, abort with a clear error (`--click caused a navigation to <url> — interact steps are for same-page UI`). Detect via a Page.frameNavigated listener armed during the interact phase.
4. `--scroll-to`: selector → `Runtime.evaluate` scrollIntoView + explicit `window.scrollTo` for a numeric y; must actually fire scroll events (real handlers listen for them).
5. Settle: after the last step, wait until two consecutive animation frames observe an identical document layout signature (lean: `Runtime.evaluate` of rAF + a cheap signature like `document.body.getBoundingClientRect()` + `scrollY` + element count), capped at 2s with the existing busy-note pattern if the cap is hit (`note: page had not settled after interactions`).
6. Combined with `--viewports`: interact steps re-run per viewport page (same pattern as v6 state re-resolution).

**Fixture `fixtures/interactive/index.html`:** pinned-size page with (a) a `button#menu-btn` whose inline-script click handler toggles a class on `#menu` making it a 400px-wide dropdown that visibly bleeds its 300px parent (bleed exists ONLY after click); (b) a scroll handler that, past y=400, adds a class rendering a 16x16 fixed "back to top" `a#top-link` (tap-target violation exists ONLY after scrolling); (c) enough pinned-height content to scroll.

**RED tests:** (a) `check` on the fixture → no violations; `check --click "#menu-btn"` → the parent-bleed with exact px; (b) `check --scroll-to 500` → the tap-target violation; (c) `layout --click "#menu-btn"` shows `#menu` at 400px, without click 0/hidden; (d) click on a selector matching nothing → suggestions error; (e) two `--click` occurrences accumulate (fixture: second click toggles the menu closed → no violations — proves ordering AND array flags); (f) MCP `check` with `click` param mirrors (a); (g) determinism: two identical `--click` runs byte-identical.

**Commit:** `Added interaction pre-steps for click and scroll states`

### Task 2: Settled capture — `--settled` and `--at-time`

**Files:** New `src/core/animate.ts`. Modify `src/cli.ts`, `src/mcp.ts` (layout/check/inspect/explain/verify get both flags; snapshot/diff get `--settled` ONLY — a settled snapshot is deterministic and desirable; `--at-time` stays excluded there). Fixture: `fixtures/animated/index.html`. Tests: new `test/animate.test.ts` + skin additions.

**Contract:**
1. `--settled`: after navigation (and any interact steps), fast-forward every animation/transition to its end state, then capture. Mechanics: `Animation.enable` BEFORE navigation (events fire as animations are created — cache them), then for each cached animation `Animation.seekAnimations` to its full duration (startDelay + duration × iterations). Infinite-iteration animations cannot end: pause them at their current time via `Animation.setPlaybackRate(0)` and append `note: N infinite animations frozen mid-flight`. Verify Animation-domain coverage of CSS transitions vs animations vs Web Animations EMPIRICALLY and document what you find — this is the task's known unknown.
2. `--at-time N` (ms): seek all cached animations to time N (clamped to their duration) and capture. Same infinite-animation note if any.
3. Ordering with other features: interact steps run first (they often START transitions — that's the point), then settle/seek, then state forcing, then capture. Document.
4. Without the flags, behavior is byte-identical to today (Animation.enable must NOT be enabled unless a flag is present).
5. `verify` gains `--settled` pass-through (agents should verify end states, but keep the default unchanged in v8 — flag it in the tool description as recommended for animated pages).

**Fixture `fixtures/animated/index.html`:** (a) an element with a CSS transition `width 200px → 400px` over 10s triggered at load (class added by inline script immediately) inside a 300px parent — mid-flight width is time-dependent, settled width is exactly 400px and bleeds 100px; (b) an infinite spinner animation (transform — layout-neutral) to exercise the freeze note.

**RED tests:** (a) `check --settled` → parent-bleed exactly 100px (without the flag the result is time-dependent — do NOT assert the unsettled case beyond "runs"); (b) `layout --settled` shows the element at 400px, and TWO settled runs are byte-identical (determinism proof — this is the money assertion); (c) `--at-time 0` shows 200px; (d) the spinner produces the infinite-animation note under `--settled`; (e) snapshot+diff round-trip with `--settled` → `(no layout changes)`; (f) `--at-time` on snapshot → loud rejection.

**Commit:** `Added settled and at-time capture for animated pages`

### Task 3: `stability` — load-time layout-shift report

**Files:** New `src/core/stability.ts`, new CLI command + 8th MCP tool. Fixture: `fixtures/shifty/index.html`. Tests: new `test/stability.test.ts` + skin additions. README: tools table row.

**Contract:**
1. `bettercss stability <url> [--duration MS] [--threshold SCORE] [--viewport WxH]` / MCP tool `stability`. Mechanics: `Page.addScriptToEvaluateOnNewDocument` installs a buffered `PerformanceObserver` for `layout-shift` entries (keep source elements via `entry.sources`, stashing element references in a global array so they survive to collection); navigate; wait `--duration` (default 3000ms) past load; collect entries via `Runtime.evaluate`; map each source element to our selector language (resolve through `DOM.describeNode`/backendNodeId machinery or an in-page `selectorOf`-equivalent — implementer's judgment, document it).
2. Output: first line `STABILITY: <cumulative score> (threshold <t>)` then one line per shift: `[+<ms>] div.hero moved (0,64)→(0,210) score 0.31`, then suspects: any `<img>`/`<video>` source element without both width and height attributes gets `suspect: img.banner has no intrinsic size attributes`. Exit 1 when score > threshold (default 0.1 — the Core Web Vitals "good" boundary).
3. This tool is an OBSERVATION, not a deterministic snapshot: say so in the tool description ("timing-dependent; local dev servers under-report — throttle to reproduce production shifts"). Determinism tests are NOT required for the score itself; the fixture is built to shift deterministically (JS setTimeout, not network) so the ELEMENT ATTRIBUTION is assertable.
4. `stability` takes no interact/state/settled flags in v8 (YAGNI until asked).

**Fixture `fixtures/shifty/index.html`:** pinned layout where an inline script inserts a 200px-tall unsized banner block after 300ms above existing content (deterministic shift, score comfortably > 0.1), plus one img-without-dimensions to exercise the suspect heuristic.

**RED tests:** (a) `stability` on the shifty fixture → exit 1, first line matches `/^STABILITY: 0\.\d+/`, output names the pushed-down content's selector and the `+300`-ish timing bucket, and includes the intrinsic-size suspect; (b) `stability` on fixtures/fluid → exit 0, `STABILITY: 0` (no shifts); (c) MCP tool listed (count 8) and mirrors (a).

**Commit:** `Added stability command reporting load-time layout shifts`
